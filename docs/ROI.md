# ROI — Sheriff aVatar 도입 효과 추정

> 수치 출처: 사내 실측 (2026-07, 실 CI 파이프라인 관측 + 당번 운영 현황). 가정은 본문에 명시.

## 현재 상태 (실측)

| 항목 | 값 |
|---|---|
| CI 실패 티켓 발생량 | **일평균 294건** |
| 티켓 1건 정밀 triage 시간 (원인 파악 → 담당 판정) | **20~30분** |
| 당번 운영 | 파트별 1명씩 **3~4명 선발, 3개월 로테이션**, 각자 **하루 1시간** 투입 |

이 숫자가 말하는 것 — **triage는 구조적으로 불가능한 게임이다**:

- 전량 정밀 triage에 필요한 시간: 294건 × 25분 ≈ **일 122시간**
- 실제 공급: 3~4명 × 1시간 = **일 3~4시간**
- → 정밀 triage 커버리지 **~3%**. 나머지 97%는 표면적 분류, 지연, 또는 담당자가 스스로 발견할 때까지 방치
- 3개월 로테이션마다 당번의 판정 노하우가 **리셋**된다 — "이 에러는 늘 저 팀 문제"라는 지식이 개인에게만 쌓이고 조직에 남지 않음

## SVP 도입 후

### 1) 커버리지: 3% → 100%
294건 전량이 발생 수 초 내 1차 분류된다 — 티켓에 링크된 Jenkins 실패 샤드에서 **해당 TC의 실행 구간 로그를 자동 첨부**하고(사람은 이걸 찾는 데 시간 대부분을 씀), vault의 과거 해결 사례를 근거로 신뢰도를 매긴다.

### 2) 당번 시간: 판정에서 확인으로
- 신뢰도 >80 → 담당자 자동 배정 (Jira assignee + 근거 댓글 + In Progress). **보수 가정 30%**로 시작 — 일 ~88건이 당번 손을 거치지 않음
- 신뢰도 ≤80 → 당번 큐. 단, **실패 로그·과거 사례·판정 근거가 첨부된 상태**라 건당 20~30분 → 5~10분으로 단축 (당번은 판정이 아니라 확인만)

### 3) 정량 추정 (보수적)

| 항목 | 계산 | 값 |
|---|---|---|
| 현재 당번 투입 | 3.5명 × 1h × 21일 | **월 ~74시간** |
| 자동 배정 30% + 확인 시간 1/3 단축 | 74 × (0.3 + 0.7 × 0.5) | **월 ~35~40시간 절감** (엔지니어 시간, 연 ~450시간) |
| 배정 리드타임 | 당번 1시간 창구 대기 (최대 1일) → 폴링 주기 | **일 단위 → 분 단위** |

자동 배정률 30%는 **시작점**이다 — 해결된 티켓이 case-log로 축적될수록(F7 ingest) 같은 유형의 신뢰도가 올라가 자동 배정률이 증가한다. 이 compounding이 규칙 기반 자동화와의 결정적 차이 (README "기존 솔루션과 무엇이 다른가").

### 4) 정성 효과
- **조직 기억**: 3개월마다 리셋되던 triage 노하우가 vault에 영구 축적 — 신규 당번 온보딩 비용 소멸, 담당자 퇴사·이동에도 지식 유지
- **증거 보존**: Jenkins 콘솔은 로테이션으로 수개월 내 소실되지만, 해결 시점에 `raw/`로 동결되어 역추적 가능
- **안전한 도입**: 3단 write 게이트(`dry-run`→`label`→`live`)로 무위험 관찰 → 부분 → 전면 전환

## 확장성 (시장 가치)

파이프라인이 사내 표준 스택(Jira + Jenkins) 위에서 동작하고 팀 고유 설정은 `.env`(JQL·호스트)뿐이다 — **다른 프로젝트/팀 복제 비용이 설정 1파일**. CI 실패 triage에 당번을 두는 조직이면 어디든 같은 구조가 적용된다.
