# SVP 데모 시나리오

> 목표: **"CI 실패 → Jira 티켓 → LLM이 WIKI를 읽고 담당자 매칭 → Jira 요약 댓글 → 해결 → WIKI가 배운다"**
> 한 사이클을 15분 안에 보여준다. 핵심 메시지는 마지막 장면 — **같은 유형의 이슈가 두 번째엔 자동 배정된다 (compounding)**.

## 등장 인물 / 장비

| 역할 | 데모 계정 | 실행 |
|---|---|---|
| 당번 (C) | carol | Sheriff Avatar **서버 모드** (대시보드) + mock Jira 서버 |
| 팀원 (A) | alice — auth 담당 | Sheriff Avatar 클라이언트 (컴팩트 창) |
| 팀원 (B) | bob — payment 담당 | Sheriff Avatar 클라이언트 (컴팩트 창) |

- 이상적으로는 PC 3대. 1대로 할 경우 창 3개를 띄우되 팝업 위치가 겹치므로 리허설로 배치 확인.
- 진행자는 터미널에서 `POST /demo/trigger`로 티켓을 생성한다 ([API.md §4](./API.md)). "사내 CI가 방금 실패했다"는 내레이션과 함께.

## 사전 준비 (데모 시작 전 체크리스트)

- [ ] `git restore wiki-vault` — 이전 테스트로 오염된 case-log/index/log 초기화
- [ ] mock Jira 기동(8792) → C 서버 모드 기동(허브 8791) → A·B 클라이언트 접속
- [ ] 세 화면 모두 "서버 연결됨" 초록 표시 확인
- [ ] `.env`에 Claude API key 유효 확인 (실패해도 fallback으로 데모는 진행되지만 신뢰도가 전부 0이 됨)
- [ ] 우하단 팝업을 가리는 다른 앱 알림 끄기 (Windows 집중 지원 켜기)

## 장면 1 — 아는 이슈는 자동 배정 (3분)

> 트리거: `POST /demo/trigger { "scenario": "auth-token-401" }`
> — "LoginFlowTest token refresh 401" 티켓. `wiki-vault/modules/auth.md`에 known-failure로 이미 존재하는 유형.

| 순서 | 화면 | 관객이 봐야 할 것 |
|---|---|---|
| 1 | C 대시보드 | 이슈 카드 등장 — **신뢰도 >80**, 근거로 `modules/auth.md` 표시 |
| 2 | mock Jira (`GET /demo/tickets`) | 티켓에 **요약 댓글**이 달리고 assignee = alice로 지정됨. 댓글의 "요약"이 로그 없이도 읽히는 것 강조 |
| 3 | A 컴팩트 창 | 우하단 팝업 → 클릭하면 이슈로 이동. **B 화면에는 아무것도 안 뜸** (서버 측 필터링) |
| 4 | A | "확인" → "해결 완료" 클릭 |
| 5 | C 대시보드 + wiki-vault | 이슈가 해결됨으로 변경, `case-log.md`에 사례가 **자동 기록**되는 것을 파일로 열어 보여줌 |

**멘트 포인트**: "LLM이 WIKI에서 과거 사례를 찾았기 때문에 사람 개입 없이 담당자까지 갔습니다."

## 장면 2 — 모르는 이슈는 당번에게 (human-in-the-loop, 4분)

> 트리거: `POST /demo/trigger { "scenario": "snapshot-diff" }`
> — "SnapshotTest ui_diff 불일치" 티켓. WIKI에 근거가 없는 처음 보는 유형.

| 순서 | 화면 | 관객이 봐야 할 것 |
|---|---|---|
| 1 | C 대시보드 | **신뢰도 ≤80** → 당번에게 배정. 요약에 "확실한 근거 없음, 당번 확인 필요" |
| 2 | mock Jira | 댓글에도 "당번 확인 필요"로 기록 — 자동화가 **아는 척하지 않는다**는 점 강조 |
| 3 | C | 당번이 이슈 내용을 읽고 → **수동 재배정**으로 bob 지정 |
| 4 | B 컴팩트 창 | 팝업 수신 → 처리 → "해결 완료" |
| 5 | wiki-vault | case-log에 이 사례가 기록됨 — **다음 장면의 복선** |

**멘트 포인트**: "모르는 건 사람에게 넘깁니다. 80점 임계값이 자동화와 사람 판단의 경계선입니다."

## 장면 3 — WIKI가 배웠다 (compounding, 3분) ★ 하이라이트

> 트리거: 장면 2와 **같은 유형** 재발 — `POST /demo/trigger { "scenario": "snapshot-diff-2" }`

| 순서 | 화면 | 관객이 봐야 할 것 |
|---|---|---|
| 1 | C 대시보드 | 같은 유형인데 이번엔 **신뢰도가 >80으로 상승** — 근거에 방금 쌓인 case-log 항목이 표시됨 |
| 2 | B 컴팩트 창 | 당번을 거치지 않고 **바로 bob에게** 팝업 |

**멘트 포인트**: "방금 해결한 기록이 WIKI에 남았고, 같은 이슈가 다시 오자 시스템이 스스로 담당자를 찾았습니다.
쓸수록 자동 배정 비율이 올라갑니다 — 이게 이 프로젝트의 존재 이유입니다."

## 장면 4 — WIKI 위생 루프 (2분, 시간 없으면 생략 가능)

| 순서 | 화면 | 관객이 봐야 할 것 |
|---|---|---|
| 1 | A | 이슈 카드의 참조 노트에 👎 (사전에 2회 눌러두고 데모에선 3회째만) |
| 2 | C | "🔍 WIKI 점검" 클릭 → 보고서에 해당 노트가 **정리 후보**로 표시 |

**멘트 포인트**: "쓸모없는 지식은 팀원 투표로 걸러집니다. WIKI는 쌓이기만 하는 게 아니라 관리됩니다."

## 마무리 (1분)

- 한 장 요약: 유입(Jira) → 분류(LLM+WIKI) → 배정(80점 라우팅) → 기록(ingest) → **다음 분류가 더 정확** — 루프 그림 ([README.md](../README.md)의 mermaid) 재사용.
- 남은 일: 사내 Jira 실연동(스키마 확정), 분류 정확도, EXE 배포 — [PLAN.md](./PLAN.md) Week 3.

## 리허설에서 자주 터지는 것

- 장면 3에서 신뢰도가 안 오르는 경우 → 장면 2의 "해결 완료"를 빼먹었거나 case-log 미기록. **장면 2의 5번을 반드시 확인하고 넘어갈 것.**
- 1대 PC 데모에서 팝업이 즉시 사라짐 → 팝업 TTL 9초. 트리거 직후 말을 멈추고 화면을 가리킬 것.
- 데모 후 `git status`에 wiki-vault 변경이 남는다 → 커밋 금지, `git restore wiki-vault` (README 병합 체크리스트와 동일 규칙).
